# PPT HTML Pipeline

Use this skill when the goal is to **produce PPT-style HTML pages** with the existing pipeline in `/root/.openclaw/workspace/tools`, especially for tasks like:
- “把这页做成 PPT 风格 HTML”
- “按任务包生成单页汇报 HTML”
- “先确认逐页详细需求稿，再批量生成每页 HTML”
- “批量做第 04–10 页这种页面”
- “复用现有 HTML 制作脚本 / 流水线”
- “把逐页详细需求稿 → blueprint → HTML → 审核 跑通”

Do **not** use this for normal websites, landing pages, admin dashboards, or generic web UI work.

## Skill vs pipeline

To avoid confusion:

- **`ppt-html-pipeline` is the skill name**
- The skill wraps an existing **underlying PPT page pipeline script**
- That underlying pipeline is what actually runs the build/review stages

So the recommended wording is:
- use the **`ppt-html-pipeline` skill**
- the skill calls the **PPT page pipeline script** underneath

Do not describe the underlying script itself as “the skill”.

## What this skill wraps

This skill packages the existing workspace pipeline, not a brand new system.

### Actual execution entry used by the skill
- `skills/ppt-html-pipeline/scripts/ppt_page_pipeline.py`
- `skills/ppt-html-pipeline/scripts/validators.py`

### Underlying shared modules / source-of-truth modules
These supporting modules are currently loaded from the workspace tools directory:
- `/root/.openclaw/workspace/tools/task_parser.py`
- `/root/.openclaw/workspace/tools/pipeline_types.py`
- `/root/.openclaw/workspace/tools/prompt_loader.py`
- `/root/.openclaw/workspace/tools/llm_client.py`

This means the skill-level pipeline script is the execution entry, but some parsing / timeout / prompt-loading / LLM access logic still comes from shared workspace modules.

### Portability requirement
This skill should be treated as **host-config-driven**, not machine-hardcoded.

That means:
- the LLM client used by the pipeline should resolve against the host OpenClaw's current configured model/provider setup
- the skill should not rely on one specific local provider name, one specific base URL, or one machine-specific API layout as a permanent contract
- prompt assets may be shipped with the skill, but model/provider resolution should come from the host environment
- by default, the pipeline should prefer the host OpenClaw default model configuration when available

If host default configuration is missing or incomplete, the implementation should fail clearly or perform a one-time local compatibility setup step before the pipeline run, rather than silently binding to one machine-local hardcoded provider.

Prompt / workflow assets that matter to the pipeline should be treated as part of the pipeline contract and must not be casually dropped when adjusting the flow.

Other original workspace source-of-truth files still live in:
- `/root/.openclaw/workspace/tools/ppt_page_pipeline.py`
- `/root/.openclaw/workspace/tools/validators.py`

## Core concepts and boundaries

Keep these layers distinct:

### 1. 逐页详细需求稿
This is the **user-aligned page planning document**, and it is also the direct pipeline input file.
It should answer things like:
- this page's role in the deck
- what the page must communicate
- previous / next page relation
- required content blocks
- style expectations
- forbidden directions
- notable layout intent

Recommended pattern:
- one markdown file per page
- pre-confirmed with the user before build
- directly used as the file passed into the pipeline

### 2. Blueprint
This is the pipeline's **single-page construction plan / 施工图阶段产物**.
It is generated by the pipeline after parsing the detailed requirement draft.
It is not the same thing as the user's original detailed requirement draft.

### 3. HTML
This is the final reviewed page output.

## Full recommended workflow

For real delivery work, the complete preferred flow is:

0. **Run preflight to ensure the host LLM client/config is ready**
1. **Confirm page-by-page detailed requirement drafts with the user**
2. **Store one requirement markdown file per page**
3. **Run the underlying PPT page pipeline once per page, using that page's detailed requirement markdown directly as input**
4. **Let the pipeline handle blueprint generation/review and HTML generation/review automatically**
5. **The controller should not participate in the normal internal pipeline stages**
6. **Only intervene manually when a stage is blocked, fails, or review does not pass**
7. **Collect final page HTML files**

In short:

`逐页详细需求稿 -> blueprint -> blueprint review -> HTML -> HTML review -> 最终单页 HTML`

## Preflight requirement

Before the pipeline starts, the skill should first check whether the host OpenClaw default model/provider configuration is ready for LLM access.

Expected preflight behavior:
- first check whether there is an already-prepared compatible LLM client result for the current host default model; if yes, reuse it directly
- if not, try to bootstrap one automatically from the current host OpenClaw default model/provider configuration
- verify the resolved provider has the minimum required client fields (for example `baseUrl`, `apiKey`, and a usable model id)
- run a lightweight smoke test against the resolved host default model
- if the environment is already ready and the smoke test passes, continue automatically
- if the environment is not ready, perform one-time compatibility setup when supported, or fail clearly with an actionable error
- if the request succeeds but the response parsing shape is incompatible, stop early and treat it as a required fix for the LLM client adapter layer
- this whole preflight/bootstrap/repair sequence should be handled inside the skill without making the user manage the client manually

This preflight is part of the skill's front-door behavior, not an optional extra.

## Default workflow

### A. Single-page build
Use when you already have a page-by-page detailed requirement markdown file.

Command pattern:
```bash
python3 /root/.openclaw/workspace/skills/ppt-html-pipeline/scripts/ppt_page_pipeline.py \
  --task-file <task-md> \
  --output-html <output-html> \
  --work-dir <tmp-dir>
```

Here, `--task-file` should point to the page's detailed requirement markdown file.

The pipeline runs in this order:
1. parse detailed requirement draft
2. generate blueprint
3. review blueprint
4. build HTML
5. review HTML
6. write final HTML

Artifacts are written into `--work-dir`, typically including:
- `task.json`
- `01-blueprint.md`
- `02-blueprint-review.md`
- `03-page.html`
- `04-html-review.md`
- `run-log.jsonl`
- `final-report.md`

### B. Batch build
If the user wants many pages built in one go, use a controller process that:
- starts from **one detailed requirement markdown per page**
- calls the single-page pipeline per page
- keeps per-page `work-dir`
- supports skip / resume / retry
- collects final per-page HTML outputs

For a concrete example, read:
- `references/batch-example.md`

## What the controller / 主控 is responsible for

The controller layer should do these things:

1. **Manage page-by-page requirement drafts**
   - ensure each page has its own md file
   - ensure they are user-confirmed before production

2. **Launch the underlying pipeline per page**
   - one run per page
   - one `work-dir` per page
   - no mixed artifacts across pages

3. **Stay out of normal internal pipeline stages**
   - if parse, blueprint review, HTML build, and HTML review all pass, the controller should not manually interfere
   - the controller is an orchestrator, not a normal participant inside every stage

4. **Recognize and surface only real exceptions**
   - blocked stage
   - pipeline failure
   - review failure
   - obvious quality drift that escaped automatic review

5. **Collect outputs and status**
   - final HTML path per page
   - failed pages list
   - retry-needed pages list
   - optional preview bundle

## Required detailed requirement draft shape

The markdown file passed to the current parser must preserve these exact field prefixes on single lines:
- `- 页码：`
- `- 页面标题：`
- `- 页面目标：`
- `- 上一页关系：`
- `- 下一页关系：`
- `- 风格要求：`
- `- 禁用项：`
- `- 核心内容：`
- `- 特殊硬规则：`
- `- 输出路径：`

Current parser behavior to be aware of:
- it only parses lines starting with `- `
- it only parses the Chinese full-width colon `：`
- it reads one line at a time; multi-line spillover is not parsed as part of the same field
- minimum hard validation currently checks `页码 / 页面标题 / 页面目标 / 输出路径`
- however, for quality, the other fields should still be filled whenever known

If these drift, parse failures or quality drift are likely.

## How to build good detailed requirement drafts

The biggest quality risk is under-specification.

A good page requirement draft should capture:
- page role in the overall deck
- required information blocks
- must-show statements
- required layout direction when known
- strong style constraints
- explicit forbidden patterns
- special hard rules from the user

Do **not** replace a rich page requirement draft with a thin summary unless the user explicitly wants rough first-pass generation.

## When manual intervention is required

Manual intervention is required when any of these happen:
- parse failure
- missing required fields
- blueprint review fails
- html review fails
- final HTML is clearly webpage-like rather than PPT-like
- final HTML ignores major page-role requirements from the detailed requirement draft
- the same failure repeats twice

If the same page fails twice, do not brute-force rerun blindly. Fix the requirement draft, validator, prompt asset, or controller logic first.

## Non-negotiable quality rules

- The page must feel like a **mature PPT slide**, not a normal webpage.
- Do not default to topbars, nav bars, stacked web cards, hero sections, or product landing page patterns.
- Prefer strong slide hierarchy, projection-friendly typography, and clear reading path.
- If the page is sparse, fix the page structure; do not hide weak content behind decoration.
- When the page is part of a reviewed multi-page deck, page role and deck continuity matter as much as local prettiness.

## Recommended file organization

A good default structure is:

```text
<project>/requirements/
  page-02-详细需求稿.md
  page-03-详细需求稿.md
  ...

  page-03-xxx.html
  ...
```

Exact paths may vary per project, but the layer separation should remain clear.

## Current implementation status

Current direction after refinement:
- model/provider resolution is expected to come from the host OpenClaw configuration
- the pipeline prefers the host default model setting first
- the LLM client now also reads provider `api` mode and adjusts request shape/path for different configured API styles instead of assuming a single protocol
- a preflight step now checks whether the host LLM client/config is ready before the pipeline starts
- the preflight also runs a lightweight smoke test and validates whether text can actually be extracted from the model response shape
- a ready client result can be cached and reused for the current resolved host default model
- response adapter selection can now be learned and cached per `provider/model` during smoke test and normal requests
- when fixed adapters do not match, the controller can now try candidate text paths mined from the real raw response structure, score candidate paths, avoid metadata-only fields like `choices[0].message.role`, and cache the successful path-based adapter
- if preflight/smoke-test parsing fails, the controller can now retry internal repair loops before surfacing a final failure
- machine-local hardcoded provider assumptions should be avoided as much as possible

Remaining automation work, if needed, should focus on:
- expanding one-time compatibility repair when the host environment is not yet ready
- broadening response-shape adapters when a new upstream/provider returns a different payload format
- improving candidate-path ranking for harder provider payloads
- broadening protocol support beyond the currently handled API styles when needed
- keeping the bootstrap/preflight logic generic across host machines
- then continuing the pipeline automatically with the resolved host default model

## Working style

When using this skill for delivery work:
1. confirm whether there is already a page-by-page detailed requirement draft; if not, create one first
2. keep one requirement draft markdown per page
3. pass those page drafts directly into the underlying pipeline
4. keep output paths explicit
5. keep `work-dir` isolated per page
6. let the pipeline run its internal stages without controller interference when all stages pass
7. inspect failures before blind retries
8. if the same failure happens twice, fix the requirement draft, validator, prompt asset, or controller instead of brute-force rerunning
9. only interrupt the user when a stage actually blocks or review fails
